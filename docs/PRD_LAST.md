## Overview

### 이 프로젝트는 무엇을 위한 것인가

GearGap은 World of Warcraft 유저의 **의사결정 피로를 완화하는 투명한 대시보드**다.

흩어진 데이터(SimC 시뮬레이션, Wowhead 드롭 정보, Blizzard 캐릭터 정보, Raider.IO 평균 클리어 시간)를 합쳐서 보여주고, 유저가 스스로 파밍 로드맵을 결정할 수 있도록 돕는다.

### 왜 하려 하는가

#### 개인적 동기

이 프로젝트는 시장 진출이 아닌, **개발자 이전부터 본인이 와우 플레이어로서 풀고 싶었던 문제**를 개발로 푸는 것이 1차 목적이다.

```
"뭘 먹어야 할지는 알아.
 Archon, Wowhead, bloodmallet 다 보면 나와있어.
 근데 그 수가 너무 많고, 내 시간은 제한적이야.

 어디부터 가야 가장 스펙업에 도움이 되지?"
```

#### 시장 검증 결과

조사 과정에서 **본인이 풀려는 문제가 실제 유저들도 동일하게 겪고 있음**이 확인됨:

- 유저들은 "시간 대비 효율"에 강한 집착이 있음 (특히 직장인 게이머)
- 흑마법사 유저들은 이미 수동으로 효율 점수 매겨 로드맵 짬
- Raidbots/bloodmallet/Wowhead 데이터를 종합하는 도구의 부재 → 엑셀로 직접 계산

다만 부정 신호도 명확히 존재:

- AskMrRobot(AMR)의 자동 추천 도구는 외면당함 → 블랙박스 알고리즘 불신
- 시뮬 데이터 자체에 대한 회의 ("패치워크 가정의 이론치일 뿐")
- 효율 추구 문화가 번아웃 유발 ("World of Simcraft" 피로감)

→ 이 부정 신호들이 GearGap의 **정체성**과 **설계 원칙**을 결정함.

## Success Metrics

본 프로젝트는 시장 진출이 1차 목적이 아니므로, 일반적인 비즈니스 지표(MAU, 매출 등)는 적용하지 않는다. 대신 **본 프로젝트의 동기와 정렬된 지표**를 사용한다.

### Phase 1 (MVP) 성공 지표

| 분류 | 지표 | 목표 |
|---|---|---|
| 기능 완성도 | 본인 흑마 캐릭터로 dogfooding 시 실제 사용 가능 여부 | YES |
| 투명성 | 모든 효율 점수에 출처/계산식이 명시되는가 | 100% |
| 데이터 정확도 | bloodmallet 데이터와 GearGap 표시 데이터 일치율 | 99% 이상 |
| 사용자 통제권 | 가중치 슬라이더 조정 시 결과 즉시 재계산 | 1초 이내 |

### Phase 2 성공 지표

| 분류 | 지표 | 목표 |
|---|---|---|
| 이행 추적 | 캐릭터 스냅샷 정확도 | 100% |
| 실측 비교 | 시뮬 예측 vs 실측 DPS 갭을 정량적으로 표시 | 구현 완료 |
| 자연어 설명 | Claude API를 통한 변화 설명의 정확성 | Mirror Agent 검토 통과 |

### Phase 3 성공 지표 (북극성)

| 분류 | 지표 | 목표 |
|---|---|---|
| 개인화 학습 | 시뮬 vs 실측 갭이 시간에 따라 줄어드는가 | 본인 데이터 N=15 이상 시 검증 |
| 신뢰도 지표 | 유저별 신뢰도 지표가 의미있게 산출되는가 | 정성 평가 |

### 메타 성공 지표

> "Mirror Agent로 본 PRD를 다시 검토했을 때, AMR 패턴(블랙박스/정답 자판기)을 발견하지 못함"

이 지표는 본 프로젝트의 **철학 유지** 여부를 검증한다.

## Messaging

### 한 줄 설명 (Elevator Pitch)

> "흩어진 와우 데이터를 합쳐 보여주는 투명한 대시보드. 도구는 정답을 주지 않고, 결정은 유저가 합니다."

### 키 메시지

| 메시지 | 의도 |
|---|---|
| "정답 자판기가 아닙니다" | AMR과의 차별화 |
| "당신의 데이터로 당신에게 맞춰갑니다" | 검증 루프(북극성) 강조 |
| "모든 숫자에는 출처가 있습니다" | 투명성 원칙 |
| "가중치는 당신의 것입니다" | 유저 통제권 |

### 사용해야 할 단어

- ✅ "참고", "후보", "효율 점수", "근거", "출처"
- ✅ "의사결정 대시보드", "통합 시각화"

### 사용 금지 단어

- ❌ "추천", "최적", "정답"
- ❌ "1위/2위/3위" (순위 강조)
- ❌ "AI가 알아서" (블랙박스 인상)

## User Scenarios

### Scenario 1: "주간 파밍 계획 짜기"

**상황:** 화요일 저녁, 정재웅은 이번 주 4시간 정도 와우를 할 수 있다. 어디 가서 뭘 먹을지 결정해야 한다.

**기존 방식:**
1. bloodmallet 열어서 트링켓별 DPS 향상도 확인
2. Wowhead에서 트링켓이 어디서 드롭되는지 확인
3. Raider.IO에서 본인 캐릭터 진행도 확인
4. 머릿속에서 "DPS 높은 거 ÷ 시간 × 확률" 계산
5. 결정 (15-30분 소요)

**GearGap 방식:**
1. GearGap 접속, 캐릭터명 입력
2. 통합 대시보드에서 후보 콘텐츠 리스트 확인
3. 가중치 슬라이더로 본인 우선순위 반영 ("나는 시간이 더 중요해" → Time 가중치 ↑)
4. 결과 확인 → 의심나는 점수는 (?) 클릭해서 근거 확인
5. 결정 (3-5분 소요)

### Scenario 2: "근거 검증하기"

**상황:** GearGap이 "한밤 일반 3넴드 = 효율 점수 13.3"으로 표시. 정재웅은 의심한다.

**유저 행동:**
1. 13.3 옆 (?) 아이콘 클릭
2. 근거 패널 확인:
   - DPS Gain 800: bloodmallet 2026-04-15 시뮬 (패치워크 기준)
   - Time 30분: Raider.IO 평균
   - Drop 50%: Wowhead 보고 데이터
   - 계산식: 800 ÷ 30 × 0.5 = 13.3
3. "패치워크 기준이라 이동 많은 보스는 다를 수 있음" 경고 확인
4. 본인 판단: "그래도 갈 만하다" → 결정

### Scenario 3 (Phase 2): "지난주 결과 확인"

**상황:** 다음 주 화요일, 정재웅은 GearGap에 다시 접속.

**유저 행동:**
1. 캐릭터 조회
2. GearGap이 자동 감지: "지난주 이후 새로 먹은 아이템: 트링켓 A, 반지 B"
3. "예측 DPS +800 / 실측 DPS +600 (75%)" 표시
4. Claude API 설명: "이번 주 영웅 레이드 시뮬 예측보다 75% 수준의 DPS가 나왔습니다. 이동이 많은 2넴드 영향일 가능성이 있습니다."
5. 다음 주 가중치 자동 보정 제안 (단, 투명하게 명시)

## User Stories / Features / Requirements

### 우선순위 매트릭스

| 우선순위 | 정의 |
|---|---|
| **P0 (필수)** | 이게 없으면 GearGap이 아님 |
| **P1 (중요)** | MVP에 반드시 포함되어야 함 |
| **P2 (유용)** | Phase 2 이상에서 추가 |
| **P3 (북극성)** | Phase 3에서 구현, 장기 비전 |

### Phase 1 기능

#### F1. 캐릭터 조회 [P0]
> "유저로서, 캐릭터명과 서버를 입력하면 내 현재 장비를 자동으로 가져오길 원한다."

- 캐릭터명 + 서버 입력
- Blizzard API 연동
- 스펙 자동 감지 (파멸/고통/악마소환)
- **왜 P0인가:** 모든 다른 기능의 입력 데이터

#### F2. 데이터 통합 시각화 [P0]
> "유저로서, 흩어진 데이터(DPS Gain, 시간, 확률)를 한 화면에서 보고 싶다."

- 후보 콘텐츠를 카드 형태로 표시
- 모든 숫자에 출처 라벨
- 모든 숫자에 한계 표시 ("⚠️ 패치워크 기준")
- **왜 P0인가:** GearGap의 핵심 가치

#### F3. 효율 점수 + 가중치 슬라이더 [P0]
> "유저로서, 효율 점수 공식의 가중치를 직접 조정하고 싶다."

- 기본 효율 점수 표시
- DPS / Time / Drop 가중치 슬라이더 (0.5~2.0)
- 실시간 재계산
- **왜 P0인가:** "가중치는 유저의 것" 철학의 구현

#### F4. 근거 패널 (Why this number?) [P0]
> "유저로서, 표시된 점수의 근거를 추적할 수 있어야 한다."

- 효율 점수 옆 (?) 아이콘
- 클릭 시 근거 모달
  - DPS Gain 출처/시뮬 시점/조건/한계
  - Time 출처/표본 크기/한계
  - Drop 출처/표본 크기/한계
  - 계산식
- **왜 P0인가:** "투명성 우선" 철학의 구현. 이게 빠지면 AMR과 동일.

#### F5. 정렬/필터 [P1]
> "유저로서, 콘텐츠를 다양한 기준으로 정렬/필터하고 싶다."

- 정렬: 효율 점수 / DPS Gain / 시간 짧은 순 / 드롭률 높은 순
- 필터: 콘텐츠 타입, 난이도, 슬롯
- **단:** "1위/2위" 표기는 절대 사용 금지
- **왜 P1인가:** UX에 중요하지만 P0 기능 없이는 의미 없음

### Phase 2 기능

#### F6. 캐릭터 스냅샷 [P2]
> "유저로서, 시점별로 내 캐릭터 상태를 저장하고 비교하고 싶다."

- 조회 시점마다 자동 스냅샷
- 이전 스냅샷과 비교 가능
- **왜 P2인가:** 검증 루프의 전제 조건

#### F7. 변화 감지 [P2]
> "유저로서, 지난번 이후 새로 먹은 아이템과 안 먹은 아이템을 알고 싶다."

- 스냅샷 비교 후 변화 표시
- "이행한 추천 / 이행 안 함" 분리
- **왜 P2인가:** "이행 추적" 단계의 핵심

#### F8. 실측 DPS 비교 [P2]
> "유저로서, 시뮬 예측과 실제 DPS의 차이를 알고 싶다."

- Warcraft Logs API 연동
- 예측 vs 실측 비교 UI
- **왜 P2인가:** 검증 루프의 데이터 수집 단계

### Phase 3 기능 (북극성)

#### F9. 개인 보정치 학습 [P3]
> "유저로서, 내 실측 데이터에 따라 시뮬 가중치가 자동 보정되길 원한다."

- 시뮬 vs 실측 갭을 시간에 걸쳐 학습
- 유저별 보정 계수 산출
- **왜 P3인가:** N≥15 이상의 데이터 누적 후 의미 있음

#### F10. 컨텍스트 인식 가중치 [P3]
> "유저로서, 내가 자주 가는 콘텐츠가 자동으로 가중치 ↑되길 원한다."

- 단, 변경 사실은 명시 (블랙박스 금지)
- "당신이 자주 갔기에 이 콘텐츠 가중치를 1.2로 자동 조정" 같은 표시
- **왜 P3인가:** Phase 2 데이터 누적이 전제

## Features Out

명시적으로 **하지 않는** 것들과 그 이유.

### FO1. 자체 시뮬 엔진

- **왜 안 하나:** SimulationCraft가 이미 있고 신뢰받음. 자체 시뮬은 AMR 실패 패턴.

## Q&A

### Q1. AskMrRobot이 망했는데, GearGap이 다를 수 있는 이유는?

**A:** 핵심은 **"정답 자판기 vs 의사결정 대시보드"** 의 차이.

| AMR | GearGap |
|---|---|
| 자체 수학 모델 | 신뢰받는 외부 데이터(bloodmallet/SimC)만 사용 |
| 결과만 던짐 | 모든 숫자에 출처/한계 명시 |
| 유저 통제권 없음 | 가중치 슬라이더로 유저가 조정 |
| 이행 후 검증 없음 | (Phase 3) 검증 루프로 본인 데이터에 맞춰감 |

GearGap은 알고리즘 신뢰가 아닌 **데이터 투명성과 유저 통제권**으로 신뢰를 만든다.

### Q2. 시뮬 데이터를 신뢰하지 않는다는 비판은?

**A:** 그 비판이 맞다. 그래서 GearGap은:

1. 시뮬 한계를 **항상 표시** ("⚠️ 패치워크 기준")
2. 시뮬은 **참고 지표**일 뿐 정답이 아님을 명시
3. (Phase 3) 본인 실측 데이터로 시뮬 가중치 자동 보정

시뮬 데이터를 맹신하지 않으면서도 활용하는 것이 GearGap의 입장.

### Q3. 효율 추구 문화가 번아웃을 유발한다는 비판은?

**A:** 그 비판도 맞다. GearGap의 입장:

- ❌ "최적 경로를 알려드립니다" (번아웃 가속)
- ✅ "당신의 시간 안에서 무엇을 할지 결정하는 걸 돕습니다" (보조)

문구 하나 차이지만, 도구의 정체성이 달라진다. 또한 "1위/2위" 강조를 금지함으로써 효율 굴레의 인상을 줄인다.

### Q4. 왜 흑마법사만 지원하나?

**A:** 두 가지 이유:

1. **본인이 흑마 메인이라 dogfooding 가능** — Phase 1 검증의 가장 빠른 길
2. **모든 직업/스펙은 다른 가치 함수가 필요** — 한 직업으로 검증한 후 확장이 합리적

Phase 3 이후 다른 직업 확장 검토.

### Q5. 자체 시뮬 안 돌리면 기술적으로 약해 보이지 않나?

**A:** Mirror Agent 셀프 검토 결과, 이건 **"포트폴리오 멋있어 보이려고"** 추가되는 복잡도 위험.

본 프로젝트의 핵심 가치는 **"우선순위 계산"** 이지 **"시뮬 정확도 향상"** 이 아니다. SimC가 이미 신뢰받는 표준이므로 그걸 활용하는 게 도메인에 맞는 선택.

### Q6. PRD가 너무 자주 바뀌었다 (v0.1 → v0.4). 안정성이 있나?

**A:** 변경의 본질을 보면 **점점 좁혀지는 방향**이다:

- v0.1 → v0.2: 도메인 정의 (넓은 추천 → 시간 효율)
- v0.2 → v0.3: 정체성 정의 (추천 도구 → 의사결정 보조)
- v0.3 → v0.4: 포맷 정리 (구조 변경, 내용 동일)

핵심 원칙은 **v0.3에서 잠겼다.** 4가지 설계 철학과 Anti-Pattern 6개는 변경되지 않을 예정.

### Q7. Phase 3 (검증 루프)는 정말 만들 수 있나?

**A:** 솔직히 어렵다. 검증 루프가 작동하려면:

- 유저가 추천을 따라야 함
- 따른 후 다시 GearGap에 와야 함
- Warcraft Logs에 본인 로그가 있어야 함

→ **Phase 1 완료 후 데이터 누적 상황 보고 결정.** Phase 3는 약속이 아니라 **북극성(방향성)** 으로 둠.

## Other Considerations

### 스코프를 나눈 방식과 이유

#### Phase 0 → 1 → 2 → 3 분리의 논리

| Phase | 목적 | 분리 이유 |
|---|---|---|
| 0 | 데이터 검증 | 막히면 프로젝트 자체가 위험. 가장 먼저 확인. |
| 1 | 투명 대시보드 | "시간 효율 시각화"가 본 프로젝트의 1차 가치. |
| 2 | 이행 추적 | Phase 3의 데이터 수집 단계. 그 자체로도 가치 있음. |
| 3 | 검증 루프 | 북극성. Phase 2 데이터 누적 없이는 의미 없음. |

ALLBLUE에서 본인이 자기에게 던진 비판:
> "Phase 3 통합결제 — Phase 1도 안 검증됐는데?"

이 패턴을 **Phase별 명확한 의존성 명시**로 회피.

### Mirror Agent와의 관계

본 PRD는 [Mirror Agent](https://github.com/woongblack/mirror-agent) 패턴으로 **세 차례 자체 검토**를 거쳤다. PRD 자체가 Mirror Agent의 작동 증명이다.

Mirror Agent의 핵심 명제:
> "타인에게 던진 비판을 자기 자신에게도 던질 수 있을 때, 비로소 혼자 개발이 가능하다"

GearGap은 그 명제의 **실행 결과물**.

### 개인 동기와 시장성의 관계

- **1차 목적:** 본인이 풀고 싶었던 문제를 개발로 푸는 것
- **부산물:** 결과적으로 의미있는 도구가 될 가능성

시장성은 평가 기준이 아니지만, 시장 검증을 통해 **방향성의 정확도**를 확인하는 도구로 활용.

### Loading King → Mirror Agent → GearGap 흐름

본 프로젝트는 본인의 사고 흐름의 연속선상에 있음:

| 프로젝트 | 핵심 명제 |
|---|---|
| Loading King | "기사의 머릿속 판단을 구조화한다" |
| Mirror Agent | "자기 자신에게 비판을 던진다" |
| **GearGap** | **"유저의 의사결정을 구조화하고, 자기에게 던진 비판으로 검증한다"** |

이 일관성이 본인 사고방식의 강점.
