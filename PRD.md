# PRD: GearGap
**WoW 의사결정 피로 완화를 위한 투명 대시보드 | Product Requirements Document**
작성일: 2026-05-02 | 작성자: 정재웅 | 버전: v0.3

---

## 🌟 북극성 (North Star)

> **"유저가 GearGap의 정보를 보고 스스로 결정한 로드맵을, 실제로 이행했을 때 그 결과가 본인에게 맞아 들어가는 경험"**

신뢰는 알고리즘의 명성에서 오지 않는다. **본인의 이행 결과로 쌓인다.**

---

## 📐 설계 철학 (Design Philosophy)

이 4가지는 **타협 불가능한 원칙**이다. 모든 기능 결정은 이 원칙에 정렬되어야 한다.

### 1. 투명성 우선 (Transparency First)
모든 숫자에는 출처와 계산식이 따라온다. 블랙박스 없음.

### 2. 도구는 정답을 주지 않는다
효율 점수는 추천이 아니라 **참고자료**다. 결정은 유저가 한다.

### 3. 가중치는 유저의 것이다
DPS / 시간 / 확률의 비중을 유저가 조정할 수 있다.

### 4. 한계를 숨기지 않는다
시뮬은 패치워크 기준임을 항상 명시한다. **실전 ≠ 시뮬.**

## 1. 배경 및 문제 정의

### 진짜 문제

> "뭘 먹어야 할지는 알아. Archon, Wowhead, bloodmallet 다 보면 나와있어.
>  근데 그 수가 너무 많고, 내 시간은 제한적이야.
>  **어디부터 가야 가장 스펙업에 도움이 되지?**"

이건 **추천 문제가 아니라 정보 통합/우선순위 시각화 문제**다.

### 시장 검증 결과

**긍정 신호:**
- 유저들은 "시간 대비 효율"에 강한 집착이 있음 (특히 직장인 게이머)
- 흑마법사 유저들은 이미 수동으로 효율 점수 매겨 로드맵 짬
- Raidbots/bloodmallet/Wowhead 데이터를 종합하는 도구의 부재 → 엑셀로 직접 계산
- 유저 언어("시간 대비 효율", "파밍 동선", "파밍 큐")와 GearGap 도메인 정의가 일치

**부정 신호 (가장 중요):**
- **AskMrRobot(AMR)의 실패 사례** — 자동 "Best in Bags" 추천이 외면당함
- 유저들은 **블랙박스 알고리즘을 신뢰하지 않음**
- 시뮬 수치 자체에 대한 회의 — "패치워크 가정의 이론치일 뿐"
- "World of Simcraft" 피로감 — 효율 추구 문화가 번아웃 유발

### AMR이 실패한 이유 (GearGap이 반복하지 말아야 할 것)

| AMR의 선택 | 결과 |
|---|---|
| 자체 수학 모델로 SimC 대체 시도 | 신뢰 상실 |
| "정답"을 즉시 제시 | 유저가 검증할 수 없음 |
| 변수 통제권을 도구가 가짐 | 유저 통제감 박탈 |

**Raidbots가 살아남은 이유:** 번거롭더라도 **유저가 직접 통제하고 검증 가능**했기 때문.

### GearGap의 위치

GearGap은 **새로운 시뮬 엔진을 만들지 않는다.**
이미 신뢰받는 도구들(SimC, bloodmallet, Wowhead, Blizzard API)의 데이터를 **합쳐서 보여주고**, **유저가 스스로 판단할 수 있도록 돕는다.**

| 도구 | 영역 | 한계 |
|---|---|---|
| Archon | 메타/티어 빌드 | 내 캐릭터 무관 |
| bloodmallet | 시뮬 시각화 (DPS Gain) | 일반 프로필 기준 |
| Raidbots | 시뮬 (내 캐릭터 입력) | "결과"만 제공, 우선순위 없음 |
| **GearGap** | **위 데이터들의 통합 + 시간 효율 시각화 + 유저 가중치 조정** | — |

## 2. 동기 (Personal Motivation)

이 프로젝트는 시장 진출이 아닌, **개발자 이전부터 본인이 풀고 싶었던 문제를 개발로 푸는 것**이 목적이다.

다만 시장 조사 결과 본인이 풀려는 문제가 **실제 유저들도 동일하게 겪고 있다는 것이 확인됨**. 시장성을 추구하지 않더라도, 결과적으로 의미있는 도구가 될 가능성이 높음.

## 3. 핵심 도메인 개념

### 효율 점수 (Efficiency Score)

> "이 콘텐츠 한 번 돌 때 시간 대비 얼마나 강해지는가에 대한 **참고 지표**"

```
Efficiency = (DPS Gain) ÷ (Time Cost) × (Drop Probability)
```

**중요:** 이 점수는 **순위가 아니라 참고 지표**다. UI에서 "1위/2위" 같은 표기는 **금지**한다.

**구성요소:**

| 요소 | 정의 | 데이터 소스 | 한계 명시 |
|---|---|---|---|
| DPS Gain | 해당 아이템 장착 시 예상 DPS 향상량 | bloodmallet (SimC 결과) | 패치워크 기준, 실전 ≠ 시뮬 |
| Time Cost | 해당 콘텐츠 1회 클리어 평균 소요 시간 | 정적 데이터 / Raider.IO | 평균값, 공대 수준에 따라 다름 |
| Drop Probability | 해당 아이템 드롭 확률 | Wowhead | 통계적 평균, 운에 좌우됨 |

### 가중치 조정 (User Weights)

유저는 효율 점수 공식의 가중치를 직접 조정할 수 있다.

```
Efficiency = (DPS Gain × W_dps) ÷ (Time Cost × W_time) × (Drop Probability × W_prob)
```

기본값은 모두 1.0이며, 유저 슬라이더로 조정 가능. 

이는 **"가장 효율적인 정답이 있다"는 가정을 부정**하는 핵심 기능. 유저마다 시간 가치, 운 vs 확정 드롭 선호, 시뮬 신뢰도가 다르기 때문.

### 검증 루프 (Verification Loop) — 북극성 기능

GearGap의 진짜 차별점이자 장기 비전.

```
T0: 로드맵 제시
    - 현재 DPS 추정치 스냅샷
    - 후보 콘텐츠 + 효율 점수
        ↓
유저가 결정 (어디 갈지)
        ↓
T0 + 7일: 다시 캐릭터 조회
    - 무엇을 먹었나? (Blizzard API로 감지)
    - 실제 어떤 던전 갔나?
    - 시뮬 예측 DPS vs 실측 DPS (Warcraft Logs)
        ↓
갭 분석 → 다음 로드맵 보정
    - "이 유저는 시뮬 대비 실전 80%"
    - "이동 많은 보스 가중치 ↓"
    - "본인 자주 가는 던전 가중치 ↑"
```

**이 루프가 GearGap의 신뢰를 만든다.**
- AMR: "우리 알고리즘 믿으세요" → 신뢰 상실
- GearGap: "당신 데이터로 당신에게 맞춰가요" → 신뢰 자가 생성

⚠️ Phase 1에서는 구현하지 않음. **북극성으로서 모든 설계 결정의 방향성**이 됨.

## 4. 스코프 (MVP)

### 콘텐츠 범위
- **레이드**: 한밤의 요새 (넴드 8개) - 일반 / 영웅 / 신화
- **쐐기 (미식)**: TWW 시즌 2 던전 풀 (8개 던전)

### 지원 직업/스펙
- MVP: **흑마법사 3개 스펙** (파멸 / 고통 / 악마소환)

### 명시적으로 하지 않을 것 (Anti-Patterns)

다음은 AMR 실패 사례에서 도출한 **금지 사항**이다.

- ❌ 자체 시뮬 엔진 개발 (SimC가 이미 신뢰받음)
- ❌ "1위/2위/3위" 같은 순위 강조 표시
- ❌ "이거 가세요" 같은 명령형 문구
- ❌ 가중치 하드코딩 (유저가 조정 가능해야 함)
- ❌ 데이터 출처 숨기기 (모든 숫자에 출처 따라가야 함)
- ❌ 시뮬 한계를 숨기기 (패치워크 기준임을 항상 명시)

## 6. 기능 요구사항

### Phase 1 (MVP) — "투명 대시보드"

**6-1. 캐릭터 조회**
- 캐릭터명 + 서버 입력
- Blizzard API로 현재 장비/스텟 수집
- 스펙 자동 감지

**6-2. 데이터 통합 시각화**
- 후보 아이템들의 DPS Gain / Time / Drop 데이터를 한 화면에 표시
- 모든 숫자에 **출처 라벨** 표시 (예: "bloodmallet, 2026-04-15")
- 모든 숫자에 **한계 표시** (예: "⚠️ 패치워크 기준")

**6-3. 효율 점수 + 가중치 조정**
- 기본 공식으로 효율 점수 표시
- DPS / Time / Drop 가중치 슬라이더 제공
- 가중치 변경 시 실시간 재계산

**6-4. 정렬/필터 (순위 강조 금지)**
- 효율 점수 순으로 정렬 가능 (단, "1위" 표기 금지)
- 콘텐츠 타입(레이드/쐐기), 난이도, 슬롯 등으로 필터
- 유저가 직접 정렬 기준을 선택

**6-5. 근거 패널 (Why this number?)**
- 각 효율 점수 클릭 시 상세 근거 표시
  - DPS Gain: "bloodmallet 2026-04-15 시뮬, 패치워크 기준"
  - Time: "Raider.IO 평균 25분 (10명 표본)"
  - Drop: "Wowhead 보고 데이터, N=2400"
  - 계산식: `800 ÷ 25 × 0.5 = 16.0`

### Phase 2 — "이행 추적 시작"

**6-6. 캐릭터 스냅샷 저장**
- 조회 시점의 장비/스텟 자동 저장

**6-7. 변화 감지**
- 재방문 시 이전 스냅샷과 비교
- "지난번 이후 새로 먹은 아이템: X, Y, Z"
- "이행한 추천: A / 이행 안 함: B"

**6-8. 실측 DPS 비교**
- Warcraft Logs API로 최근 로그 조회
- 시뮬 예측 vs 실측 DPS 표시
- "이번 주 시뮬 예측 +800, 실측 +600 (75%)"

### Phase 3 — "검증 루프 학습" (북극성)

**6-9. 개인 보정치 학습**
- 시뮬 예측 vs 실측 갭을 시간에 걸쳐 학습
- 유저별 보정 계수 자동 산출
- "당신의 시뮬 신뢰도: 82% (N=15회)"

**6-10. 컨텍스트 인식 추천**
- 유저가 자주 가는 콘텐츠 가중치 자동 ↑
- 유저가 잘하는 보스 타입(이동/패치워크) 자동 감지
- 단, 변경 사실은 명시 (블랙박스 금지)

## 8. 시스템 흐름 (Phase 1)

```
[유저] 캐릭터명 + 서버 입력
    ↓
[Blizzard API] 현재 장비 + 스텟 수집
    ↓
[스펙 감지] 파멸 / 고통 / 악마소환
    ↓
[데이터 통합] bloodmallet + Wowhead + 시간 데이터
    ↓
[효율 엔진] 가중치 적용 효율 점수 산출
    ↓
[투명 시각화] 모든 숫자 + 출처 + 한계 표시
    ↓
[유저 조정] 가중치 슬라이더로 재계산
    ↓
[유저가 결정] (도구는 결정하지 않음)
```

## 11. Mirror Agent 셀프 검토 변천 기록

본 PRD는 v0.1 작성 후 Mirror Agent 패턴으로 **세 차례** 자체 검토를 거쳤다. 각 단계에서 발견된 맹점이 다음 버전에 반영되었다.

### v0.1의 맹점 → v0.2에서 수정

## 12. 마무리

GearGap은 시장 진출이 아닌 **본인이 와우 플레이어로서 풀고 싶었던 문제**를 개발로 푸는 프로젝트다.

세 번의 자체 검토를 거치며 정체성이 명확해졌다:

- ❌ 정답 자판기 (AMR 실패 패턴)
- ✅ **투명 의사결정 대시보드**
- ✅ **장기적으로 본인 이행 데이터로 신뢰가 자라는 도구**

Loading King이 "기사가 직접 결정한 순서대로 짐을 적재하는 시스템"이었듯,
GearGap은 **"유저가 직접 결정한 로드맵을 검증해가는 시스템"** 이다.

도구는 정답을 주지 않는다. 정보를 투명하게 정리해서, 유저가 스스로 결정하고, 그 결정의 결과로 신뢰가 쌓이는 구조 — 이것이 GearGap이 AMR과 다를 수 있는 유일한 길이다.

**문제 정의가 먼저, 기술은 그 다음.** 본 프로젝트의 모든 결정은 이 원칙과 4가지 설계 철학에 정렬된다.
